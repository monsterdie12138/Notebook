    程序员不太关注到cache

    cache的好处是随用随取，但想要访问的数据是否一定在cache上，毕竟cache的大小是有限的。

    当要访问的数据在cache上，hit，否则miss。希望尽可能hit，但难免会有miss，因此导致penalty，也就是再从上一层拿数据。

    从attacker角度来说，拿得慢的在memory和storage，拿得快的在cache。

    block,line run

    访问的数据以什么为单位？

    在cache上有一个固定的单位，称为block，（内存以page为单位）。

    虚拟内存和物理内存的关联

    以block为基本单位进行调用，设计者决定一个block的大小，会影响到cache的性能，例如block小，数量就多，查找更多时间，会有balance。

    icache和dcache的分离

    safety非人为，而是数据自身是否稳定；security是外部行为攻击时，防御怎么样。

    cache的局部性：时间&空间，空间局部性->同一个block里挨着

    cache miss:

    会花费掉一些时间（响应时间），嵌入式设备要求响应时间很快，除了响应时间之外还有带宽

    引起miss的原因：

    一开始是空的，访问什么都会miss，compulsory miss，这时候需要去memory拿数据

    跟容量有关，大小有限，影响缺失率

    冲突：可能在memory中不同位置对应cache同一块，如果设计一种方法，不一定映射到特定的位置（没满就放），这样可以减少冲突，但不好找，有一个balance。

    实际上在改进性能的时候，还是在依赖局部性（时空）

    • desktop：关注平均响应时间
    • server：对带宽有较大需求
    • 嵌入式：形态多，跟app有关，快速响应时间

    36个术语：重要

    当前我们更关注cache和memory之间的关系

    我们常在L1层cache分为i和d，见图。

    cache的大小最直接的计算方法是block大小乘上数量，但这只是理论上的计算方法，实际上比计算结果大，因为每个block还有索引、是否有效之类的东西，只是比较小，所以忽略不记。这也是为什么不分太多的block。

    访问cache时：

    例如想要访问xn，左图没有，因此去上一级找，然后把xn放到cache中，这时对应右图。

    cache四问：

    • 存放问题，对应memory哪一块，映射规则（有三种方法，本质上是一种）
    • 识别问题，怎么找，用block的tag，看是否hit
    • 替换问题，如果cache满了，怎么替换
    • 写回问题，写的时候hit会怎么样，miss会怎么样，在这一章只涉及单核

    Q1：Block Placement

    假设memory也是以block为单位，如图，cache只有8块，但memory有16块，可以考虑取模，能够精确地指出mem中的某一块对应cache中的哪一块。称为direct mapped，只要知道mem的地址，就能知道对应cache的哪一块。好处是好找，原则是循环分配，问题是会存在冲突，对空间的利用率不高

    能否物尽其用，于是有Fully-associative，只要cache有位置就随便放，但是当我想要找到mem中对应到cache哪一个block时，需要遍历cache，其实实现起来最复杂（没有公式）。

    这种的方法是2-way set-associative，8个block分成四组，每组2块，mem计算公式对应某个组，但组内第几个随意。另外两种方法是这种方法的两个特殊形式。问题是每个组的大小n取多少呢？

    4-way set-associative，n太小冲突多，n太多不好找

    直接映射的替换没有选择，因为那个位置就一个block，要替换只能替换那个，剩下两种有选择。

    通常情况下每一组的n<=4

    Q2: Block Identification

    每个block都有tag，tag属于额外信息，有的还需要更多的索引

    在cache中还有valid位，只有valid位为1且mem地址和tag对应上才能hit（跟BST很像）

    tag和index共同组成cache的block的地址

    查找时分三种情况，对应映射的三种情况

    tag的大小与分组和block数相关，如果cache分为L1，L2，……，那么每一级的tag都要比较
