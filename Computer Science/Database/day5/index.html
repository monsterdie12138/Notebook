    Advanced SQL

    两块主要内容：从高级语言访问/执行SQL语句

    SQL语句的函数与过程

    SQL必须与程序语言结合，用户交互，输入输出等功能SQL没有，所以经常需要程序语言与SQL交互。

    SQL语句处理数据的力度和高级语言的是不匹配的，二者需要达到力度的匹配。程序语言设计时，每次处理一个变量，基本处理单位；但SQL涉及到的是一个relation，这才是它的基本处理单位。

    要实现这样的交互功能，有两类：dynamic SQL：在程序运行时做一组函数调用，运行时连接数据库，提供更新需求、查询需求等，数据库再把结果返回给高级程序语言。embedded SQL：在编译的时候，将SQL语句直接编译成函数调用，写高级语言的时候就表明哪几行是SQL语句。

    dynamic SQL使用ODBC（c++），JDBC（java）等接口，课上以JDBC为例。

    JDBC和ODBC是API，API是应用程序接口，对数据库服务器进行交互，JDBC和java配套使用，ODBC和c++配套使用。

    交互的时候的任务：成功与数据库建立连接，如数据库ID、端口、用户名与密码等；通过程序语言发送一些SQL查询/更新需求；结果（relation）需要一个一个的取，以元组为单位，读入程序语言的变量中。

    metadata元数据，描述数据的数据，数据库中的表格、视图、索引等，都是元数据。JDBC支持相关查询。高级语言若要访问一个不太熟悉的数据库，需要了解一下有哪些relation，名字叫什么，有哪些属性等。

    以JDBC为例，与数据库的交互可以进一步细化：

    • 打开数据库连接
    • 创造statement对象
    • 通过statement对象执行SQL语句
    • 处理问题

    Chapter6 Database Design Using the E-R Model

    接下来的两章内容角色从用户转换成设计者，如何去设计一个数据库。

    ERmodel：实体关系模型（entity-relationship）

    design phases:

    分三个阶段：

    初始阶段：各式各样的用户对数据的需求有哪些，需求调研，软件开发的固有过程

    第二个阶段：选择数据模型（概念层面的）

    最后阶段：把前面的schema转换成可以实现的数据库，一个是逻辑的实现（决定schema），一个是物理的实现（布局）。业务层面：需要记录哪些属性，计算机层面：有哪些表，每个表对应哪些属性（焦点）

    两个主要陷阱： 冗余：表中、表间信息重复，不可避免，要有个度，重复过多会有不一致的风险，存储压力也大； 不完整性：设计一个数据库，schema要尽量满足用户需求，如果漏掉一些信息，或者很难建模出来，则会有不完整性

    除了避免bad design，还要考虑如何衡量数据库的好坏

    ER图：

    很流行的建模方法

    实体（集合）+关系（集合）+属性

    一个实体是一个对象，比如一个人，一个公司，一门课程等，实体集合是共同类型属性的实体组成的，学生组成集合，老师组成集合等

    属性的子集是主键，在这里概念是一致的，主键是一个属性的集合，可以唯一标识集合中的每一个元素，与之前的定义一致。

    实体集合在ER图中表示为一个矩形，下划线为主键

    例子中学生实体和教师实体之间有一个指导关系，指导关系（advisor）是一个关系集合，关系集合指的是两个或两个以上实体构成的数据关系，用菱形表示，见图

    关系集合还可以加上属性，比如指导关系集合可以有日期属性，表示方法是矩形加虚线

    role:

    一个关系涉及到两个及以上实体，这些实体可以属于不同实体集合，也可以属于同一实体集合，例如先修课的关系，这个时候就有角色的区分，标注在线上。

    degree:

    刚刚我们遇到的都是二元关系，比较常见，度为2，也有三元关系，但三元关系也可以表示成多个二元关系，只是更好解释罢了，比较少见，例如：学生、导师、项目
