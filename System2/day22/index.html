    回顾：什么是scheduling？ 决定谁用cpu，用多久。

    scheduling的policy：不同的算法

    context switch是dispatcher（调度员）的一部分

    先到先做算法：

    gantt chart（必考！！！！！）

    waiting time，turnaround time

    convoy effect：快进程如果在慢进程后面，则总体会被延缓

    可能的改进思路：短的放前面，也就是

    shortest-job-first:

    对gantt图的解释：P1先到，先做，然后在p1做的时候剩下三个都到了，等到p1做完后，剩下的三个因为都到了，所以从最短的开始做。

    还有另一种 shortest-remaining-time-first

    waiting time怎么算：从开始到结束的时间-burst time

    短进程优先被证明是最优的算法，但问题是理论的，实际不可能实现，因为在执行一个任务之前不可能知道要执行多长时间。

    因此我们需要预测，公式见ppt，最多只能拟合

    Round-Robin：轮询调度

    是preemptive的，给每个任务分片，按序执行，对应running到ready的第二种情况

    例：分片为4，看例子应该很直接，记得如果一个任务剩下需要的时间占不满一个分片，因此会提前结束

    平均等待时间比sjf差，但响应时间response time更好（即任务都会被做，不会被插队），而sjf如果一个任务前不断插入更短的任务，则做不到了

    关键点：怎么选时间片的长短？

    如果短：疯狂的进程切换，但有不错的响应时间

    如果长：相反

    假如linux用到mobile端和server端，则前者时间片用短的，不然要等一个进程很久才能到另一个进程，会卡，即需要好的响应时间；后者时间片用长的。

    Priority Scheduling：

    人为给每一个进程分配优先级

    priority越小优先级越高（linux），但windows相反

    优先+轮询：

    注意：先优先，再轮询，意思是优先级高的肯定在优先级低的之前跑完

    优先级算法的问题：starvation：优先级低的永远被抢占，永远跑不到

    优先级最低的是0号进程（pid0），叫IDLE进程（空闲进程），为了简化scheduling的算法

    解决方法：priority aging 等的越久优先级提高越多

    Multilevel Queue Scheduling：

    缝合怪，根据优先级分成四个队列

    队内调度和队外调度

    Multilevel Feedback Queues：

    前者的改良版

    一个进程在队列之间也能移动

    吃的时间片越多，优先级降得越多

    即不怎么吃cpu的进程优先级要高，因为通常是即时交互的进程

    最general

    多核：

    单队列：调度简单，但冲突多（要上锁）

    多队列：相反

    hyperthreading

    两个线程交错着跑，由硬件维护

    Operating System Examples

    windows的调度：

    如果时间片用完，则该进程优先级降低

    linux调度：

    nice command

    如果一个进程越nice，就越容易把cpu让给其他进程，其优先级较低（被欺负）
