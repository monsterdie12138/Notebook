    data hazards

    指令间的数据依赖性

    解决方法：forwarding，冲突发生的条件→多路选择器的控制信号

    前一条指令的目的操作数和后一条指令的源操作数冲突

    EX/MEM.rd=ID/EX.rs1(rs2) MEM/WB.rd=ID/EX.rs1(rs2)

    冲突检测1：上一条指令有写回（例如其他没有冲突的指令b和j，其译码出的立即数也可能恰好相等引发冲突信号） rd不是x0（无效冲突）

    SourceA= * 情况一：来自寄存器（No forwarding） * 情况二：来自上一条指令的ALU结果（阶段间寄存器）（Forwarding with data hazard in EX/MEM） * 情况三：来自内存（Forwarding with data hazard in MEM/WB）

    双重数据冲突：

    add x1,x1,x2 add x1,x1,x3 add x1,x1,x4

    第三条指令和第一、二条指令都有冲突，但第三条指令只想要第二条指令（最近）的结果

    因此需要对冲突检测增加条件：出现MEM/WB冲突的同时不能还有EX/MEM冲突

    MEM冲突判断：

    if (MEM/WB.RegWrite && MEM/WB.rd!=x0)完整见PPT

    有关load的数据冲突：需要stall（将要stall的指令当前所在阶段后一个阶段寄存器置零）

    判断条件更改：ID/EX.rd==IF/ID.rs1(rs2) && ID/EX.MemRead（对应load指令）

    原因：再思考一下

    此时不需要判断rd为x0（load指令不会这样做），也不用判断RedWrite（load一直为0）

    软硬协同：stall时不一定要补一个bubble，可以把接下来某一个完全独立的指令先拿过来执行一下，防止影响性能效率

    控制冲突

    B型，J型，以有条件跳转（B型）为例，地址的计算是滞后的

    判别在EX，地址确定在MEM?

    解决方法：在ID阶段增加新的硬件：目标地址计算器（加法器），寄存器比较器

    如果刚好又遇到数据冲突，由于在ID就在用，所以之前的数据冲突解决方法太迟了

    实际上解决之后还是需要一个stall，只是减少了stall的数量

    现代处理器提升性能的关键部分：预测（只有在预测出错的情况下才会有惩罚，预测精度越高，惩罚越少）
